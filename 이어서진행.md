## 1. 프로젝트 한 줄 요약

네이버 플레이스 리뷰를 수집·분석해 소상공인에게 **알림 / AI답글 / 리포트 / 비교 / AI비서**를 제공하는 B2B SaaS (3인 학생팀, 12주 MVP).
요구사항 원천은 `docs/TECH_SPEC.md`. 작업 단위는 `docs/TASKS.md`(T-01~T-15, T-F1~T-F7).

## 2. 스택 (실제 구현 기준)

- 백엔드: Python **3.12 목표**(현재 개발 venv는 3.14), FastAPI, SQLAlchemy 2.0 async + asyncpg, APScheduler(워커), Playwright(크롤러)
- DB: PostgreSQL 15 + pgvector (로컬은 docker), 마이그레이션은 `backend/migrations/*.sql`
- **LLM: Google Gemini 무료 티어** (원 스펙은 Anthropic이나 유료 회피 위해 교체 — `backend/app/llm/client.py` 한 파일만 프로바이더 의존)
- 프론트: Next.js 15 App Router + TS + **Tailwind v4** + shadcn 스타일 자체 UI 컴포넌트 + **cult-ui 이식**(AnimatedNumber 등, motion) + Recharts + lucide-react + TanStack Query

15-2. **업종 확장은 v2 과제** (2026-07-24): stores.category 는 자유 텍스트라 옷가게·미용실 등록 자체는 가능(온보딩에서 칩 선택). 그러나 **분석 aspect 가 음식점 고정**(`llm/schemas.py` Literal: 맛/친절/청결/대기시간/가격/분위기/기타)이라 비음식 업종은 "맛" 항목이 어색함. 통계·대시보드·비교는 aspect 문자열에 무관하게 동작하므로, 확장 시 **LLM 분류 스키마의 category Literal 만 업종별 세트로 교체**하면 됨(예: 리테일=품질/사이즈/응대/가격/재고). 온보딩에 안내 문구 표시해둠.
15. **테스트는 데모 시드가 든 DB에서도 통과해야 한다** (2026-07-24): `run_digest`가 DB 전체를 훑기 때문에 시드가 있으면 "매장 수" 총계 단언이 깨진다(3 vs 1). → `test_run_digest_groups_and_dedup`을 **자기 매장 기준 검증**으로 수정. 앞으로 전역 스캔 함수를 테스트할 땐 총계 대신 대상 store_id 로 좁힐 것.

## 5-1. 멘토링 피드백 (2026-07-23) — 제품 방향, 코드 아님

> 스펙(`docs/TECH_SPEC.md`)은 읽기 전용이라 수정하지 않았다. MVP 기능 범위는 그대로 두고, 아래는 **발표/사업화 논의용 메모**.

- 지적: 리뷰 관리는 비용 절감(스트레스↓)이라 **유료 전환 근거가 약하다** → 매출로 이어지는 기능 필요. 기계적 답글이면 역효과. 가게 분위기별 톤이 달라야 함. 리뷰 제공에 동의하는 사장님 확보가 최대 난관. 무료/유료 차별화와 페르소나 정의 필요.
- 검토한 발전 방향(우선순위): ① 리뷰→SNS 홍보 카드 자동 생성 ② 답글에 재방문 유도(쿠폰/신메뉴) 심기 ③ 가게 페르소나 답글(+최근 답글과 유사도 비교로 반복 방지, 임베딩 재활용) ④ **무료 "우리 가게 리뷰 성적표"**(가입 없이 1회 진단 → 유입 장치) ⑤ **상권 벤치마크**(동네·업종 순위 — 쓰는 가게가 늘수록 정확해지는 네트워크 효과).
- 반론도 기록: 홍보 기능은 불황에 제일 먼저 잘리는 예산이라, 포지션을 "홍보 툴"이 아니라 **"운영 진단 툴"**(주문 누락·요일별 웨이팅·신메뉴 반응 = 매출 하락 조기 경보)로 잡는 편이 방어적이다.
- 무료/유료 안: 무료=수집+긴급 알림+답글 N건("지키는 것"), 유료=SNS 콘텐츠+경쟁 비교+주간 리포트+AI 비서("버는 것").
- 남은 숙제(코딩 아님): 사장님 5명 인터뷰 — "리뷰 얼마나 보나 / 답글 다나 / 안 단다면 왜 / 리뷰 때문에 뭘 바꿔봤나". 크롤링 법적 리스크는 **본인 동의 + 본인 가게 데이터**(CSV 임포트 경로 활용)로 회피.

### 추가 비판(2026-07-24)과 반박 논리 — 발표 Q&A 대비

**비판**: 배민·네이버가 AI 답글을 무료 내장 → 외부 SaaS로 팔면 구조적으로 불리. 답글 API 도 없어서 크롤링/자동화는 약관 위반 + 계정 정지 리스크를 사장님에게 전가. "데모는 되지만 사업화가 막히는 패턴."

**답변 구조** (인정 → 반박 → 데이터 접근 3단계):
1. **인정**: "AI 답글 생성" 자체는 커모디티 — 상품이 아니라 미끼/편의 기능. 우리도 그렇게 판단해서 포지셔닝을 "운영 진단 툴"로 옮김(위 메모). 자동 게시는 애초에 금지 설계(핵심 규칙 7 — 승인→클립보드 반자동)라 계정 정지 리스크 전가 구조가 아님.
2. **반박 — 플랫폼 내장 AI가 구조적으로 못 하는 것**: ① **채널 횡단 통합**(배민은 네이버 리뷰를 영원히 안 보여줌 — 사장님은 네이버+배민+요기요+쿠팡이츠+구글을 동시 운영) ② **독립 진단**(플랫폼 유인은 광고 판매지, "광고 말고 예약제가 답"이라는 진단이 아님) ③ **경쟁·상권 비교**(제3자만 가능, 데이터 쌓일수록 해자). 한 줄: "답글로 경쟁하지 않는다. 상품은 플랫폼이 구조적으로 못 하는 채널 횡단 평판 진단."
3. **데이터 접근 솔직하게**: 지금=본인 동의+본인 다운로드 데이터(CSV, 구현됨) / 다음=동의 파일럿 매장 한정 수집(리스크 고지) / 사업화=공식 제휴·대행 라이선스 필요함을 인정(확보 시 그 자체가 진입장벽).
4. **인터뷰 질문 업데이트**: "배민/네이버 AI 답글 써봤나, 왜 (안) 쓰나" + "여러 앱 리뷰 통합 + 원인 진단에 월 얼마까지 낼 의향" — 이 답이 이 비판의 최종 검증.

## 6. 로컬 실행 방법

```bash
# 1) DB (마이그레이션 자동 적용은 빈 볼륨 최초 1회. 이후엔 아래 6-2 참고)

uvicorn app.main:app --reload --port 8010  # http://localhost:8010/healthz (이 PC는 8000을 타 프로젝트가 점유 — 결정 13)
pytest                         # 전체 테스트
python -m scripts.seed_demo    # ★ 데모 시드 (매장2+경쟁2+리뷰366+집계+리포트, GEMINI 키 불필요)

# 3) 워커(선택)
python -m app.worker
```
